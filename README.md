# SmartParking — прогнозирование загруженности городских парковок

> **SmartParking Forecast** - учебный проект, предназначенный лишь для обучения конкретным [технологиям](#стек).

**Тип проекта:** Data Engineering + ML (end‑to‑end)  
**Статус:** MVP (`v0.1.0-mvp`)  
**Домен:** Умный город, транспортная инфраструктура  

### 1. Бизнес-контекст и проблема
В центре города водители тратят до 20–30 минут на поиск свободного места в часы пик, создавая пробки и лишние выбросы. Муниципалитет (заказчик) хочет снизить время поиска на 30% и повысить оборачиваемость парковочных мест.  
**Решение:** Cервис, который на основе исторических данных о въездах/выездах предсказывает процент заполненности каждой парковки на каждый час завтрашнего дня, и показывает водителям простой список адресов с цветовой индикацией.

### 2. Как работает система (кратко)
Каждую пятницу ночью запускается конвейер:
- Генерируются свежие синтетические логи за неделю.
- Данные проходят очистку и обогащение в dbt.
- Переобучается ML-модель (LightGBM).
- Формируется PDF-отчёт для администратора.
Пользователь (водитель) открывает веб-приложение, выбирает интересующий час, видит список парковок, отсортированный от свободных к заполненным, и принимает решение о маршруте.

### 3. Архитектура
- **Источники:** синтетический генератор (`historical_generator.py` / `weekly_generator.py`).
- **Хранилище:** PostgreSQL (отдельные БД `airflow` и `warehouse`).
- **ETL/ELT:** Apache Airflow (оркестрация), dbt (трансформации внутри БД).
- **ML:** LightGBM (регрессия процента заполненности).
- **Веб-приложение:** Streamlit (без карты, только таблицы и графики).
- **Отчёты:** ReportLab (PDF).
- **Инфраструктура:** Docker Compose (4 сервиса: postgres, warehouse, airflow‑webserver+scheduler, streamlit).

Поток данных:  
`OLTP (parkings) → Airflow DAG → dbt (raw→stage→analytics) → ML-скрипт → Streamlit / PDF`

### 4. Данные
- **16 парковочных зон** в 4 районах города (Центральный, Северный, Южный, Заречный).
- Каждая зона: адрес (может отсутствовать), этажность (1–4), максимальная вместимость (200–5000 мест).
- **Историческая таблица `parkings`:** события `entrance`/`exit` с меткой времени, этажом, тарифом.
- **3 года синтетических логов** (~50 000 записей) с реалистичными паттернами: утренние/вечерние пики, день недели, сезонность.
- **Грязь:** сиротские выезды (0.5%), дубли (0.3%), отрицательная длительность парковки (0.2%), пропуски целых дней (6 дней за период).

### 5. ETL и оркестрация
4 DAG’а Airflow, запускаемых последовательно:
1. `generate_weekly_data` (каждую пятницу 18:00) — генерация данных за прошедшую неделю.
2. `dbt_run` (после успеха первого) — запуск dbt‑моделей: очистка, интерполяция, построение витрины признаков.
3. `train_model` (после успеха второго) — переобучение LightGBM и сохранение модели.
4. `generate_report` (после успеха третьего) — создание PDF‑отчёта.

dbt‑проект содержит слои:
- `raw` – неизменённые данные.
- `staging` – очистка, удаление дублей, заполнение пропусков.
- `analytics` – витрина `fct_forecast_features` с лагами, скользящими средними, флагами дня недели и праздников.

### 6. Модель прогнозирования
- **Алгоритм:** LightGBM Regressor.
- **Целевая переменная:** `occupancy_pct` (процент занятых мест) на каждый час для каждой зоны.
- **Горизонт:** 24 часа.
- **Оценка качества:** MAE, MAPE (цель < 15%), точность цветовой классификации «зелёный/жёлтый/красный».
- **Переобучение:** еженедельное, вместе с генерацией новых данных.

### 7. Пользовательский интерфейс (Streamlit)
Две вкладки:
- **«Водитель»** – выбор часа, таблица с адресами и цветовым индикатором (зелёный ≤50%, жёлтый ≤80%, красный >80%), сортировка по заполненности.
- **«Администратор»** – график прогноза по конкретной парковке, метрики модели, кнопка «Скачать PDF».

Без карты; MVP — только список адресов.

### 8. Отчётность
PDF формируется автоматически после переобучения модели. Содержит:
- Дату формирования.
- График прогноза заполненности на следующую неделю.
- Таблицу с пиковыми часами и зонами риска (загрузка > 80%).

### 9. Бизнес-метрики
- Среднее время поиска места (ожидаемое снижение).
- Оборачиваемость одного парковочного места (машин/место/день).
- Заполненность в часы пик.

### 10. Ограничения MVP
- Только синтетические данные (реальные датчики не подключены).
- 16 статических парковок.
- Нет геокарты, только список адресов.
- Прогноз обновляется раз в сутки, не в реальном времени (butching).
- Одно окружение (отсутствует CI/CD, staging).

### 11. Версионирование и этапы
- **Недели 1–2:** `v0.1.0-alpha` – генератор, DAG загрузки, структура БД, dbt‑модели.
- **Неделя 3:** `v0.1.0-beta` – модель LightGBM, Streamlit‑дашборд, метрики.
- **Неделя 4:** `v0.1.0-mvp` – PDF‑отчёт, итоговое тестирование, документация.
